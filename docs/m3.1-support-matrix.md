# M3.1 Support Matrix: Context-aware Rules

This document defines the strict scope and constraints for the `if:` condition in security rules.

## Design Principles
1. **Context-Only**: Conditions evaluate only against the `Context` object. They cannot access or parse the tool `payload`.
2. **AOT Compilation**: Expressions are parsed and validated during policy load. If any rule has an invalid `if:` expression, the entire policy fails to load.
3. **Restricted DSL**: No functions, no arrays, no side-effects. Only basic boolean logic and string comparisons.

## Supported Variables (Whitelist)
All variables are treated as **Strings**.

| Variable | Source Field | Description |
|---|---|---|
| `actor` | `Context.actor` | The user or service account name. |
| `agent_id` | `Context.agent_id` | Unique ID of the agent instance. |
| `session_id` | `Context.session_id` | Current session ID. |
| `trust_level` | `Context.trust_level` | `untrusted`, `trusted`, or `admin`. |
| `tool` | `Tool` | Tool name (e.g., `bash`, `read_file`). |
| `working_directory` | `Context.working_directory` | Effective workspace path. |

## Supported Operators

| Operator | Type | Description |
|---|---|---|
| `==` | Comparison | Equality. |
| `!=` | Comparison | Inequality. |
| `&&` | Logic | AND. |
| `||` | Logic | OR. |
| `!` | Logic | NOT. |

## Explicitly NOT Supported
- **Functions**: `to_lower()`, `regex_match()`, etc. are prohibited.
- **Arrays/Lists**: `['a', 'b']` syntax is not supported.
- **In Operator**: `val in [...]` is not supported (use `||` logic instead).
- **Payload Access**: `payload.path` or similar fields are inaccessible.

## Fail-fast Errors (Load-time)
The following will cause `Policy::from_yaml()` or `Guard::from_yaml()` to return an `Err`:

1. **Syntax Errors**: Invalid expression syntax (e.g., `actor == 'bot' &&`).
2. **Unknown Variables**: Referencing a variable not in the whitelist (e.g., `user_email == '...'`).
3. **Disallowed Features**: Use of function calls, assignments, arrays, or loops.

## Audit Specification
When a rule is triggered via a condition:
- `matched_rule` will contain the rule index/path.
- `audit_event.details` will include:
    - `condition`: The raw condition string.
    - `condition_result`: Boolean (always true if rule matched).
