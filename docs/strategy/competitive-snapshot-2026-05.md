# Competitive Snapshot — 2026-05

> **One-time snapshot, not maintained.** Captures the state of adjacent / competing
> projects as of 2026-05-09 to inform `agent-guard` strategic direction. Re-snapshot
> if the picture changes materially; do not edit this file in place.

## Why this exists

`agent-guard` is positioning itself around **outbound change control for AI coding
agents — both action and content** (see `project_strategic_direction_2026_05`). To
do that honestly we need to know who else is in the neighborhood. This document is
the evidence base; the README and product copy are the consequence.

## Categories

The space breaks into six bands. Each band has a different layer of the stack and a
different buyer. `agent-guard` overlaps with several but is identical to none.

### 1. LLM I/O Guardrails

Filter prompts and completions, not execution.

| Project | Scope | Activity (2025-2026) | Where it ends |
|---|---|---|---|
| NeMo Guardrails (NVIDIA) | Conversational rails, jailbreak detection | Active, declining momentum vs 2024 | Stops at LLM boundary; no FS / shell / network |
| Guardrails AI | Output validation via XML schemas | Active OSS, commercial pivot | Same — content not execution |
| Llama Guard (Meta) | Safety classifier model | Versioned releases | Classifier only, no enforcement layer |
| Lakera Guard | API gateway for prompts | Acquired by Check Point 2025, integrating into CloudGuard WAF | API layer, ~100 ms latency, doesn't see filesystem / DNS / shell |
| Protect AI / Rebuff | Prompt injection detection (heuristics + embeddings) | Active | Narrow — same layer, different attack surface |

**Verdict:** orthogonal to `agent-guard`. Operate at LLM I/O layer; can coexist as
"guardrails wrap the model, agent-guard wraps the side effects."

### 2. Cloud / Ephemeral Sandbox Runtimes

Run agents inside isolated VMs / containers. Different layer.

| Project | Scope | Activity | Where it ends |
|---|---|---|---|
| E2B | Cloud sandboxes for agents, ~150-2000 ms cold start | 200M+ sandboxes shipped (claim) | Cloud only, charges per sandbox-second |
| Modal | Generic serverless + sandboxes | Active, well-funded | Cloud; not focused on permission policy |
| Daytona | Agent dev environments | Active | Cloud workspace, not local agent |
| Sprites (Fly.io), OpenSandbox (Alibaba), Northflank | Persistent / long-lived agent VMs | Emerging | Same band — cloud microVM |

**Verdict:** orthogonal. They give an agent a clean room. `agent-guard` decides
whether the agent is allowed to *do something specific* inside whatever room it
already lives in. A user could run `agent-guard` inside an E2B sandbox.

### 3. Native Agent Runtime Permission Systems

The closest band. Each major coding-agent runtime ships its own permission model.

| Runtime | What it has | Known gap |
|---|---|---|
| **Claude Code** | `allow / deny / ask` rules + PreToolUse / PostToolUse hooks + OS sandboxing (Seatbelt / bubblewrap / Job Object) | **#33106**: PreToolUse hooks have known enforcement bypass for MCP tools, closed unplanned. Sandbox can be disabled by Claude in fallback. Permission fatigue → users resort to `--dangerously-skip-permissions`. |
| **Cursor** | Background agent + per-tool permissions | Different policy dialect from Claude Code; no shared format |
| **Cline** | Auto-approve / manual rules | Yet another dialect; no cross-tool policy |
| **Codex CLI** (OpenAI) | Mostly default-allow with approval prompts | Minimal policy surface |

**Verdict:** **fragmentation is the gap**. Solo devs running ≥2 of these tools
duplicate policy work. Each runtime independently has shortcomings (Anthropic
publicly ack'd #33106). `agent-guard` is the only project I can find that aims to
be one policy across runtimes via Claude Code plugin (Sprint 8) → Cursor
(Sprint 9+) → Codex CLI.

### 4. AI-Specific Local Guards (Emerging)

The most direct overlap. Small, narrow scope each.

| Project | Approach | Scope | Verdict |
|---|---|---|---|
| Agent Shield (OSS) | Governance + config scanning | Static config audit, not runtime gating | Different layer (pre-deploy) |
| yolo-cage (OSS) | Docker + VM wrapper for Claude Code, HTTP proxy + git mediation | Blocks exfil via heavy isolation | Adds friction (VM overhead); incompatible with native shell access |
| Agent Safehouse (macOS) | Dynamic Seatbelt policy generation | Single-platform, single-runtime | Closest in spirit but macOS-only |

**Verdict:** the band where `agent-guard` directly competes. Differentiation:
cross-platform (Seatbelt + seccomp + Job Object + AppContainer all shipped) +
cross-runtime (Node + Python + Rust bindings) + cryptographic audit
(Ed25519 ExecutionProof) + zero-friction (no VM, no proxy required).

### 5. Content / PII Filtering

Adjacent — solves part of `agent-guard`'s content layer.

| Project | Scope | Activity | Where it ends |
|---|---|---|---|
| Presidio (Microsoft) | PII detection + redaction in text | Mature, active | Library; not integrated as agent runtime hook |
| OpenAI Privacy Filter | Local-runnable PII detection model | 2025 release | Model; no agent integration |
| Various secret-scanners (gitleaks, trufflehog) | Repo / file scanning | Mature | Static, not runtime |

**Verdict:** **integration gap**. Presidio is the obvious building block but no one
has wired it into a PreToolUse hook so that an agent's tool *output* (file read
result, API response) gets scanned before the LLM sees it. This is `agent-guard`
Sprint 6. Can adopt Presidio's approach (or Rust-port the patterns) without
reinventing PII detection.

### 6. Big-Cloud Guardrails

Mentioned for completeness; different buyer.

| Project | Scope | Why not relevant for our wedge |
|---|---|---|
| AWS Bedrock Guardrails | Safety / PII filters at Bedrock API | Cloud-only, locked to Bedrock-hosted models, enterprise sales |
| Azure AI Content Safety | Content filters for Azure OpenAI | Cloud-only, enterprise SKU |

**Verdict:** **structurally cannot serve the indie / solo dev local wedge**.
Customers there want "no data leaves my machine"; these products are predicated on
data leaving. This is the durable "big company won't" half of the moat thesis.

## Where `agent-guard` fits

```
┌─ LLM I/O guardrails (band 1) ──── filter prompts/completions
│
├─ Sandbox runtimes (band 2) ────── give agent a clean room (cloud)
│
├─ Native permission systems (band 3) ─ each runtime, fragmented
│
├─ AI-specific local guards (band 4) ── direct overlap, all narrow
│         ↑
│         └─ AGENT-GUARD positions here, with two unique angles:
│             1. cross-runtime (Claude Code → Cursor → Codex CLI)
│             2. cross-layer (action + content, not just action)
│
├─ Content / PII (band 5) ────────── building blocks, no agent runtime hook
│
└─ Big-cloud guardrails (band 6) ─── different buyer, different layer
```

Two-layer thesis:

- **Action layer**: gate the moment an agent tries to `git push` / `npm publish` /
  `docker push` / non-local POST / `rm -rf`. Bands 3 and 4 attempt this; nobody
  has shipped a unified, cross-runtime, signed-receipt implementation.
- **Content layer**: scan the payload before it reaches the LLM provider /
  external API. Band 5 has the building blocks (Presidio); nobody wired it up as
  a PreToolUse hook.

## Gaps that nobody currently fills

| Gap | Why it persists | Why agent-guard can | Sprint |
|---|---|---|---|
| Unified policy across coding-agent runtimes | Each vendor optimizes their own; cross-cutting work has no owner | Existing cross-platform sandbox + bindings already done | Sprint 8 (Claude Code first) → 9+ |
| PII / credentials redaction wired into PreToolUse | Presidio etc. exist as libraries, not integrations | Existing validators pipeline + Guard::run() entry | Sprint 6 |
| Tamper-evident execution receipts for compliance | Most products audit at API layer (band 1), miss filesystem / DNS | Ed25519 ExecutionProof already shipped | Sprint 6 (`verify` CLI) |
| DNS-layer exfiltration forensics ("Little Snitch for AI agents") | Requires OS-level network interception; nobody has built it for agents | Rust + libpnet feasible on macOS / Linux; Windows hard | Sprint 10+ (after PMF) |
| MCP tool response validation (closes #33106 truly) | MCP is still evolving | Doable but premature; wait for protocol stabilization | Sprint 9+ |

## Regulatory backdrop (2025-2026)

Not a competitor but a forcing function. Reproduced here so the snapshot is
self-contained:

- **EU AI Act**: full enforcement 2026-08-02 (4 months out from this snapshot).
  Agentic AI flagged as higher-risk in 2025 guidelines. Audit-trail and
  PII-non-leakage are becoming table stakes.
- **GDPR**: EUR 1.2B in 2025 fines, 73% of European AI agent implementations have
  compliance gaps per industry surveys.
- **US states**: Colorado AI Act enforcement delayed to 2026-06 then stayed by
  court 2026-04. California SB 53 (Frontier AI) requires risk frameworks +
  incident reporting, fines up to USD 1M.

**Read:** EU AI Act articles 28-31 (transparency / auditability) is the
single strongest "why now" for the cryptographic audit story.

## Skeptic check — where the gap might be smaller than it looks

Worth flagging honestly:

- **Claude Code may close #33106**: if Anthropic ships native MCP-aware
  PreToolUse hooks, the "we do what Claude Code can't" story weakens. Mitigation:
  the cross-runtime story (Cursor / Codex CLI) survives regardless.
- **Cursor / Codex CLI might unify on MCP**: protocol convergence could erode the
  fragmentation argument. Mitigation: even with shared protocol, *policy*
  fragmentation persists (each runtime decides its own defaults).
- **Presidio may add agent-runtime adapters**: would shrink Sprint 6 differentiation.
  Mitigation: integration + signed receipts + cross-runtime is still the bundle;
  Presidio alone is a library, not a product.
- **E2B / cloud sandbox vendors may push policy down**: unlikely (they monetize
  compute, not gating) but worth watching.
