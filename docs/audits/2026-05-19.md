## security-bounty-hunter

All four findings confirmed. Here's the report.

### Findings (security-bounty-hunter)

- **CRITICAL** validator bypass — variable-assignment prefix syntax escapes every blocklist
  attacker input: `Tool::Bash` with `{"command":"IGNORED= tee /etc/passwd"}` (or `FOO=1 rm -rf /workspace`, `BASH_ENV=/workspace/x.sh bash`)
  reachability: `crates/agent-guard-validators/src/bash.rs:149-234` (`check_command_segment`) and `:624-691` (`write_targets_for_segment`) both treat `parts[0]` (or `segment[command_index]`) as a literal command name. Bash variable-assignment prefix (`VAR=VAL command args…`) is shell syntax, not a command, so the prefix token (`IGNORED=`) becomes `first_command`. It is not in `WRITE_COMMANDS`, `STATE_MODIFYING_COMMANDS`, or `INLINE_CODE_INTERPRETERS`, and the `match command { … }` arm of `write_targets_for_segment` does not match it either, so neither the read-only blocklist nor `validate_paths` ever inspects the actual command at index 1. `DANGEROUS_ENV_VAR_PREFIXES` only covers `LD_PRELOAD=` / `DYLD_INSERT_LIBRARIES=` / `PYTHONPATH=` / `NODE_OPTIONS=`, so an unlisted prefix (`IGNORED=`, `FOO=`, `BASH_ENV=`, `IFS=…`) passes the token-prefix scan. Distinct from the 2026-05-18 `env`/`xargs`/`time` finding because no command name appears — the prefix is parsed by bash itself.
  impact: Full escape of ReadOnly and WorkspaceWrite path/command enforcement: `IGNORED= rm -rf /workspace/data` in ReadOnly and `IGNORED= tee /etc/passwd` in WorkspaceWrite both return `Allow`, so the underlying sandbox (or the default Noop sandbox) executes the destructive write that the validator advertises as blocked.
  remediation: In `check_command_segment` and `write_targets_for_segment`, strip any leading tokens that match the bash assignment-prefix grammar (`[A-Za-z_][A-Za-z0-9_]*=…`) before selecting `first_command` / `command`, recursing into the next token exactly like the existing `sudo` unwrap.

- **CRITICAL** path traversal — `$VAR`-prefixed redirection targets are silently skipped
  attacker input: `Tool::Bash` with `{"command":"echo data > $HOME/.ssh/authorized_keys"}` (or `cat < $HOME/.aws/credentials`, `echo x > ${HOME}/.bashrc`)
  reachability: `crates/agent-guard-validators/src/bash.rs:518` and `:544` — for every collected write/read target, `if candidate.starts_with('$') { continue; }`. This is reached from `Tool::Bash` in WorkspaceWrite mode (and ReadOnly for reads). The `contains_command_substitution` gate only rejects `$(`, backticks, and `<(`/`>(` adjacency; bare `$HOME`, `${HOME}`, `$PATH` survive intact. Bash then expands the variable at execution time and the redirection lands on a host path. The accompanying unit test at `crates/agent-guard-validators/src/tests.rs:452-458` (`dollar_prefix_skipped`) locks this in as expected behaviour, so the bypass is intentional design that ships in a workspace-confinement product.
  impact: Arbitrary out-of-workspace file write and read using only standard env vars: an attacker overwrites `$HOME/.ssh/authorized_keys` (SSH backdoor), `$HOME/.bashrc` (shell-startup persistence), or reads `$HOME/.aws/credentials`, while the validator returns `Allow` and the audit log records a "workspace-clean" write. NoopSandbox (the default when no platform sandbox is enabled) executes verbatim; Linux seccomp WorkspaceWrite adds no path filter; only macOS Seatbelt blocks it via `(allow file-write* (subpath workspace))`.
  remediation: Treat any redirection target that contains an unexpanded `$` (or `${`) as **deny**, not skip — `validate_paths` cannot reason about expansion and the safe default is fail-closed; provide a `safe_env_vars` allowlist if legitimate use cases need it.

- **HIGH** SSRF — IPv4-mapped/compatible IPv6 addresses bypass the executor deny-list
  attacker input: `Tool::HttpRequest` with `{"method":"POST","url":"http://[::ffff:127.0.0.1]:8080/admin"}` (also works for `::ffff:10.0.0.1`, `::ffff:169.254.169.254`, `::127.0.0.1`)
  reachability: `crates/agent-guard-sdk/src/executors.rs:164-194` (`is_always_blocked_ip`) calls `ip.is_loopback()` which, for `IpAddr::V6`, dispatches to `Ipv6Addr::is_loopback()` — that stdlib helper compares strictly against `::1`, so the IPv4-mapped loopback form `::ffff:127.0.0.1` returns `false`. The IPv6 branch then checks `is_unspecified | is_multicast | is_ipv6_link_local | is_ipv6_unique_local`; none cover the `::ffff:0:0/96` mapped range. `resolve_url_to_safe_addr` accepts the literal IPv6 host, pins reqwest to that `SocketAddr::V6`, and Linux/macOS dual-stack TCP transparently routes the mapped address to the IPv4 loopback. The 2026-05-15 patch (1a339da) added `::1` and `fc00::/7` but missed mapped/compatible forms.
  impact: Direct reach to localhost-bound services (admin APIs, cloud-metadata neighbours, internal RPCs) and to the *full* RFC1918 / link-local range via `::ffff:10.0.0.1`, `::ffff:169.254.169.254`, etc., from the "safe" mutation HTTP executor that the SDK markets as SSRF-hardened. Mutation methods give POST/PUT/PATCH/DELETE to those endpoints, not just read access.
  remediation: After resolution, additionally reject any `IpAddr::V6(v6)` whose first 80 bits are zero followed by either `0x0000` (IPv4-compatible) or `0xffff` (IPv4-mapped), then re-run `is_always_blocked_ip` on the embedded IPv4 — or convert the address to its canonical `Ipv4Addr` via `v6.to_ipv4_mapped()` / `to_ipv4()` and recurse.

- **HIGH** validator bypass — stdin-fed interpreter laundering evades `INLINE_CODE_FLAGS`
  attacker input: `Tool::Bash` with `{"command":"bash <<< 'rm /workspace/important.txt'"}` (or `echo 'rm file.txt' | bash`, `sh < /workspace/.staged_script`)
  reachability: `crates/agent-guard-validators/src/bash.rs:217-232` — interpreter laundering is only detected when an `INLINE_CODE_INTERPRETERS` first-token is followed by an arg in `INLINE_CODE_FLAGS = ["-c", "-e", "-r", "--command", "--exec"]` (line 93). The list does not cover stdin-fed forms: `<<<` here-string, `<` input redirection, and pipe-to-interpreter (`… | bash`) all hand the interpreter executable code via fd-0 instead of a flag. The `<<<` form survives `shell_split` as a single token (not in `INLINE_CODE_FLAGS`); the pipe form puts `bash` alone in a downstream segment where `parts[1..]` is empty so the flag scan trivially passes. `validate_paths` only inspects `<` redirection targets; `<<<` and `|` carry no path target so the workspace-boundary check never fires on the literal payload either.
  impact: Full validator bypass equivalent to `bash -c "<arbitrary>"` in ReadOnly mode. Chained with the destructive-pattern gap (`bash <<< 'rm file.txt'` lower-cased contains no `rm -rf /` substring), the call returns `Allow` rather than even `Warn`/`AskUser`, so the sandbox executes attacker code that the audit record claims is a `bash` invocation against no scripts.
  remediation: In `check_command_segment`, when `first_command` is an `INLINE_CODE_INTERPRETERS` member, also block if (a) any arg is `<<<` or `<`, or (b) the segment is the right-hand side of a `|` and has no script-file argument — treating "interpreter reads stdin" as equivalent to "interpreter with `-c`".


## type-design-analyzer

### Findings (type-design-analyzer)

- **HIGH** `CustomToolId` (`crates/agent-guard-core/src/types.rs`) — `#[derive(Deserialize)]` on the newtype bypasses the `new()` constructor, allowing invalid or builtin-shadowing IDs to be deserialized without triggering length, charset, or conflict checks.
  ```rust
  // A crafted JSON payload produces CustomToolId("bash") — rejected by new() but
  // accepted silently by the derived Deserialize impl:
  let id: CustomToolId = serde_json::from_str("\"bash\"").unwrap();
  // id.as_str() == "bash" — conflicts with the builtin Tool::Bash name
  ```
  remediation: Replace `#[derive(Deserialize)]` with a manual impl (or `#[serde(try_from = "String")]`) that routes through `CustomToolId::new()` and propagates `CustomToolIdError`.

- **MEDIUM** `GuardDecision` (`crates/agent-guard-core/src/decision.rs`) — `#[derive(Deserialize)]` allows any caller to construct `GuardDecision::Allow` from a string literal without going through policy evaluation, creating a forged-decision surface for any downstream consumer that deserializes and trusts this type.
  ```rust
  let d: GuardDecision = serde_json::from_str(r#"{"decision":"allow"}"#).unwrap();
  assert!(d.is_allowed()); // true — no policy check occurred
  ```
  remediation: If `Deserialize` is only needed for audit-log round-trip reads, scope it to a separate `AuditedDecision` read-only type; remove `Deserialize` from the enforcement-facing `GuardDecision`.

- **MEDIUM** `TrustLevel` / `Context` (`crates/agent-guard-core/src/types.rs`) — `TrustLevel::Admin` (and `Trusted`) can be reached by direct struct literal construction or via `Deserialize` with no opt-in gate; `Context` is fully `pub` with no constructor, so `Context { trust_level: TrustLevel::Admin, .. }` requires zero ceremony.
  ```rust
  let input = GuardInput::new(Tool::Bash, r#"{"command":"rm -rf /"}"#)
      .with_context(Context { trust_level: TrustLevel::Admin, ..Default::default() });
  ```
  remediation: Make `Context::trust_level` private and expose a builder that requires explicit justification for non-`Untrusted` levels, or add an audit gate in the SDK's `check_internal` path when `Admin` trust is asserted.

Note: `DecisionCode` and `PolicyMode` reviewed — no exploitable findings (`PolicyMode::default()` is `ReadOnly`, `TrustLevel::default()` is `Untrusted`, both safe).
