### Findings (silent-failure-hunter)

- **CRITICAL** `crates/agent-guard-sdk/src/anomaly.rs:126` — `report_denial` silently returns when the `states` mutex is poisoned, preventing the deny-fuse counter from incrementing and allowing a misbehaving agent to exhaust its denial budget without ever being locked out.
  remediation: Replace the silent `return` with a `tracing::error!` and fail closed by treating every in-flight call as a denial or by locking the subject immediately, matching the fail-closed posture that `check()` already applies on line 63.

- **HIGH** `crates/agent-guard-sdk/src/executors.rs:237-245` — `payload_declares_mutation_http` uses `.ok().unwrap_or(false)` so a malformed or truncated `HttpRequest` payload silently returns `false`, routing the call to `RuntimeDecision::Handoff` instead of SDK-owned sandboxed `Execute`, bypassing the execution sandbox without any log or error.
  remediation: Return an explicit error or log the parse failure and treat an unparseable payload as `Execute` (fail closed toward SDK ownership) rather than silently handing off to the host.

- **HIGH** `crates/agent-guard-sdk/src/provenance.rs:38` — `SystemTime::now().duration_since(UNIX_EPOCH).unwrap_or(0)` silently stamps execution receipts with a Unix epoch timestamp (1970-01-01) on any clock error, making the signed receipt's timestamp field meaningless and weakening tamper-detection based on timestamp ordering.
  remediation: Return an error from `ExecutionReceipt::sign` instead of silently inserting a zero timestamp, so callers know signing failed rather than receiving a structurally valid but semantically broken receipt.

- **HIGH** `crates/agent-guard-sdk/src/audit_writer.rs:91` — `thread::Builder::new().spawn(...).expect("failed to spawn audit-writer thread")` panics and crashes the process if the OS cannot spawn the worker thread, rather than returning the error as a `GuardInitError`.
  remediation: Change `expect()` to `map_err(...)` and surface the failure as `GuardInitError::AuditFileOpen` or a new `AuditWriterSpawn` variant so callers can handle it gracefully without a process crash.

- **HIGH** `crates/agent-guard-sandbox/src/windows_appcontainer.rs:262` — `let _ = GetExitCodeProcess(pi.hProcess, &mut exit_code)` silently discards the Win32 API failure; if `GetExitCodeProcess` fails the local `exit_code` remains `0`, making a failed sandboxed process appear to have succeeded.
  remediation: Check the return value and return a `SandboxError::ExecutionFailed` if `GetExitCodeProcess` fails, matching the pattern used in `windows.rs` for the same call.

- **MEDIUM** `crates/agent-guard-sdk/src/enforce.rs:384-386` — `serde_json::to_string(&record).unwrap_or_else(|e| format!(...))` in `report_handoff_result` writes a synthetic error-sentinel JSON blob to the audit file on serialization failure without emitting any `tracing::error!` log, making the serialization failure invisible unless someone is actively parsing the audit file.
  remediation: Add a `tracing::error!` call inside the `unwrap_or_else` closure so the failure is observable in the log stream even if the fallback record is still written.

- **MEDIUM** `crates/agent-guard-sdk/src/executors.rs:43` — `serde_json::from_str(payload).map_err(|_| SandboxError::ExecutionFailed("invalid payload JSON".to_string()))` discards the actual parse error (field name, line, column) in `execute_write_file`, making it impossible to diagnose why a payload was rejected.
  remediation: Replace `|_|` with `|e|` and include `e.to_string()` in the `ExecutionFailed` message, identical fix applies to the same pattern on line 82 in `execute_http_request`.

- **MEDIUM** `crates/agent-guard-sandbox/src/windows_appcontainer.rs:265-266` — `t1.join().unwrap_or_default()` and `t2.join().unwrap_or_default()` silently replace stdout/stderr with empty strings if the reader threads panic, hiding any output-capture failure in sandboxed execution results.
  remediation: Use `unwrap_or_else(|_| "[stdout read error]".to_string())` with a `tracing::error!` to preserve observability, matching the explicit fallback string pattern already used in `windows.rs:607-610`.

- **MEDIUM** `crates/agent-guard-sandbox/src/windows_appcontainer.rs:311` — `let _ = SetHandleInformation(read_pipe, 1, 0)` silently ignores failure to clear the `HANDLE_FLAG_INHERIT` bit on the read end of the pipe, potentially allowing the sandboxed child process to inherit the parent's read handle and read its own stdout.
  remediation: Check the return value and return `SandboxError::ExecutionFailed` on failure, matching the identical check already present in `windows.rs:334-340`.
