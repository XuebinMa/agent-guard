## silent-failure-hunter

### Findings (silent-failure-hunter)

- **CRITICAL** `crates/agent-guard-sdk/src/executors.rs:237` — `payload_declares_mutation_http` calls `.ok()` on JSON parse failure and returns `false`, so a malformed `HttpRequest` payload silently resolves to `RuntimeDecision::Handoff` instead of `Execute`, bypassing the DNS-block validation (`resolve_url_to_safe_addr`) that guards against SSRF.
  remediation: Return an error or treat parse failure as a block/deny rather than defaulting to `false`; `runtime_decision_for_input` should have an explicit error arm for unparseable HTTP payloads.

- **HIGH** `crates/agent-guard-sdk/src/anomaly.rs:126` — `report_denial` silently returns on a poisoned mutex while `check()` at line 63 returns `AnomalyStatus::Locked` on the same condition; the asymmetry means a denial that should have incremented the Deny Fuse counter is lost, so the fuse may never trip if the mutex is poisoned during each denial.
  remediation: Log the poison at `error!` level and unify handling with `check` so fuse state and check state agree under poison.

- **HIGH** `crates/agent-guard-sdk/src/provenance.rs:36` — `SystemTime::now().duration_since(UNIX_EPOCH).unwrap_or(0)` silently places a zero timestamp in a signed execution receipt; a receipt with `timestamp: 0` is signed and returned as valid, making it structurally indistinguishable from a legitimate receipt.
  remediation: Change `ExecutionReceipt::sign` to return `Result` and fail if the system clock is not ahead of the epoch instead of signing a zero timestamp.

- **HIGH** `crates/agent-guard-sandbox/src/windows_appcontainer.rs:262` — `let _ = GetExitCodeProcess(pi.hProcess, &mut exit_code)` silently swallows a Win32 failure; on error `exit_code` stays 0, causing the sandbox to report success for a process whose actual exit status is unknown.
  remediation: Check the return value and propagate `SandboxError::ExecutionFailed` with `GetLastError` if the call fails.

- **HIGH** `crates/agent-guard-sandbox/src/windows_appcontainer.rs:311` — `let _ = SetHandleInformation(read_pipe, 1, 0)` silently ignores failure to clear the inherit flag on the pipe read-end; child processes can then inherit the read-end of stdout/stderr pipes, a handle leak that may also deadlock the child.
  remediation: Check the return value and return `SandboxError::ExecutionFailed` if `SetHandleInformation` fails.

- **MEDIUM** `crates/agent-guard-sandbox/src/windows_appcontainer.rs:265` — `t1.join().unwrap_or_default()` / `t2.join().unwrap_or_default()` silently converts a panicked stdout/stderr reader thread into an empty string without logging, hiding I/O thread panics from the audit record.
  remediation: Match the `Err` arm of `join()` and emit `tracing::error!` or propagate `SandboxError::ExecutionFailed`.

- **MEDIUM** `crates/agent-guard-sandbox/src/windows.rs:608` — `stdout_thread.join().unwrap_or_else(|_| "[Error reading stdout]".to_string())` swallows reader-thread panics and stores a sentinel string in `SandboxOutput::stdout` without logging or returning an error.
  remediation: Log or propagate panics from the reader threads rather than silently substituting a sentinel string.


## security-bounty-hunter

### Findings (security-bounty-hunter)

- **HIGH** validator bypass — `env`/`xargs`/`find -exec`/`time` laundering past ReadOnly blocklist

  attacker input: `env rm -rf /workspace/important` (or `xargs rm < /workspace/filelist`, `find /workspace -exec rm {} \;`, `time rm /workspace/secret`, `nice rm ...`, `nohup rm ...`)

  reachability: External caller submits `Tool::Bash` in `ReadOnly` mode. `check_command_segment` recursively unwraps only `sudo`; common POSIX wrappers (`env`, `time`, `nice`, `nohup`, `xargs`, `find -exec`, `flock`, `strace`, `watch`, `nsenter`) are not recognized. `first_command` becomes `env` etc., which is in none of `WRITE_COMMANDS`, `STATE_MODIFYING_COMMANDS`, or `INLINE_CODE_INTERPRETERS`, so the check returns `None` (allow). The destructive sub-command then executes (with `NoopSandbox` fallback giving full workspace mutation).

  impact: Arbitrary write/delete in a session the policy declares ReadOnly, breaking the core mode guarantee.

  remediation: Extend `check_command_segment`'s recursive-unwrap (currently `sudo`-only) to also unwrap `env`, `time`, `nice`, `nohup`, `xargs`, and `find` when `-exec` is present, then re-evaluate the unwrapped sub-command as the effective first command.

- **HIGH** validator bypass — bare `exec` built-in escapes command-level checks

  attacker input: `exec /bin/sh` (or `exec env LD_PRELOAD=/tmp/evil.so cat /etc/shadow`)

  reachability: `Tool::Bash` payload in any non-Blocked mode. The `exec` shell built-in is not in any blocklist (`WRITE_COMMANDS`, `STATE_MODIFYING_COMMANDS`, `INLINE_CODE_INTERPRETERS`), so `check_command_segment` returns `None`. `exec` replaces the current shell process with the named binary, escaping the validator entirely.

  impact: Full shell escape on `NoopSandbox`; partial mitigation by Seatbelt/seccomp on hardened platforms but still a clear validator-layer bypass.

  remediation: Add `exec`, `eval`, `source`, and the `.` built-in to `STATE_MODIFYING_COMMANDS` (or a new "process-replacing built-ins" category).

- **MEDIUM** policy DSL — condition-only rule with no match criterion fires unconditionally

  attacker input (policy YAML):
  ```yaml
  tools:
    bash:
      allow:
        - if: "trust_level == \"untrusted\""
  ```

  reachability: Anyone who can write/reload policy YAML (multi-tenant policy frags, writable policy file, reload endpoint). In `pattern_matches` (policy.rs ~L710-712), when `if` evaluates true and no `prefix`/`regex`/`plain` is supplied, `result.matched = true`, so the rule allows every payload whenever the condition holds.

  impact: All deny rules for the affected tool are bypassed for matching contexts — agent can self-allow arbitrary bash/file/HTTP.

  remediation: Reject `allow` rules that contain only `if` and no concrete match criterion at policy-compile time (fail-closed); only permit condition-only rules in `deny`/`ask` lists.

- **MEDIUM** signature verification not enforced — `Unsigned` executes identically to `Verified`

  attacker input: Load Guard via `Guard::from_yaml(attacker_controlled_yaml)` (no signature attached).

  reachability: Callers using `from_yaml` / `from_yaml_file` / `reload_from_yaml` set `PolicyVerification::unsigned()`. `should_fail_closed()` only returns `true` for `Invalid`, not `Unsigned`, so an attacker-supplied allow-all unsigned policy runs with full permissions. The Ed25519 signing infrastructure exists but the SDK has no mode that requires signatures.

  impact: Attacker who can land a policy file (FS write, API injection, env var) bypasses any org-level signing requirement.

  remediation: Add `Guard::require_signed()` (or `PolicyVerificationMode::RequireSigned`) that causes `execute()`/`run()` to fail closed on `Unsigned`, not only `Invalid`; document as the recommended production setting.


## type-design-analyzer

### Findings (type-design-analyzer)

- **CRITICAL** `GuardDecision` (`crates/agent-guard-core/src/decision.rs`) — `Deserialize` is derived unconditionally, so any code path that deserializes a `GuardDecision` from external input (stored audit log, webhook payload, IPC) can receive `{"decision":"allow"}` and produce `GuardDecision::Allow` without any policy evaluation.
  ```rust
  let d: GuardDecision = serde_json::from_str(r#"{"decision":"allow"}"#).unwrap();
  assert!(d.is_allowed()); // true — no policy was consulted
  ```
  remediation: Split the type into a write-side (`GuardDecision`, no `Deserialize`) and a read-side audit DTO that derives `Deserialize`; or add a custom `Deserialize` impl that rejects the `Allow` variant when reading from external sources.

- **HIGH** `DecisionReason` (`crates/agent-guard-core/src/decision.rs`) — all four fields (`code`, `message`, `details`, `matched_rule`) are `pub`, so any caller holding a mutable binding (e.g. after destructuring a `Deny` variant) can overwrite the `code` or zero out the `message`, corrupting the audit trail after the decision is emitted.
  ```rust
  if let GuardDecision::Deny { mut reason } = guard.check_tool(...) {
      reason.code = DecisionCode::InternalError; // silently reclassifies the denial
      reason.message = String::new();            // erases the audit message
  }
  ```
  remediation: Make the fields `pub(crate)` and expose only immutable getters; use the existing builder methods as the sole mutation surface.

- **HIGH** `PolicyMode` / `PolicyFile` (`crates/agent-guard-core/src/policy.rs`) — when YAML sets `default_mode: full_access` and omits `trust.untrusted`, `Untrusted` callers receive `FullAccess` because `effective_mode` falls back to `default_mode`; the type system places no floor on what `Untrusted` can inherit, so a misconfigured policy silently grants full execution rights to the lowest trust tier.
  ```yaml
  version: 1
  default_mode: full_access
  # No trust: block — Untrusted now gets FullAccess
  ```
  remediation: Cap `Untrusted` effective mode at `WorkspaceWrite` unconditionally in `effective_mode`, or introduce a distinct `UntrustedMode` newtype that cannot represent `FullAccess`.

- **HIGH** `DecisionCode` (`crates/agent-guard-core/src/decision.rs`) — the enum is not marked `#[non_exhaustive]`, so downstream `match`/`matches!` gates written today silently fall through future deny-class variants, potentially treating a new deny code as non-deny.
  ```rust
  fn is_hard_deny(code: &DecisionCode) -> bool {
      matches!(code, DecisionCode::PathTraversal | DecisionCode::AgentLocked /* ... */)
      // future DecisionCode::NetworkBlocked silently returns false
  }
  ```
  remediation: Add `#[non_exhaustive]` so downstream exhaustive matches break at compile time when new variants are added.

- **MEDIUM** `TrustLevel` / `Context` (`crates/agent-guard-core/src/types.rs`) — `Context` derives `Default` (correctly `Untrusted`) but all fields including `trust_level` are `pub`, so any code holding `&mut Context` can escalate to `Admin` post-construction without any access-control check.
  ```rust
  let mut ctx = Context::default();           // Untrusted
  ctx.trust_level = TrustLevel::Admin;        // silently escalated
  guard.check_tool(tool, payload, ctx);       // evaluated at Admin
  ```
  remediation: Make `Context` fields `pub(crate)` and expose construction through a builder/constructor that requires an authenticated principal to claim a non-default trust level.
