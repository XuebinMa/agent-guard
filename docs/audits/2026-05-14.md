### Findings (security-bounty-hunter)

- **CRITICAL** validator-bypass — Bash command substitution `$(...)` / backticks slip past read-only and path checks
  attacker input: `{"tool":"Bash","command":"echo data > $(echo /etc/passwd)"}` (WorkspaceWrite); or `{"command":"echo `rm -rf /workspace/data`"}` (ReadOnly)
  reachability: `crates/agent-guard-validators/src/bash.rs::shell_split` treats `$(`, `)`, and `` ` `` as ordinary characters, so the inner command is never re-validated. `validate_paths` then sees the redirection target as `$(echo /etc/passwd)`, hits the `candidate.starts_with('$')` skip at line 368, and returns `Allow`. The sandbox executes via `Command::new("sh").arg("-c").arg(command)` (e.g. `crates/agent-guard-sandbox/src/noop.rs:37-43`, also linux/macos paths), so the shell performs the substitution at run time.
  impact: Arbitrary writes outside the workspace and arbitrary write/destructive-command execution while the policy mode advertises ReadOnly / WorkspaceWrite — full validator bypass of the Bash wedge.
  remediation: Reject (or recursively re-validate) any command containing `$(`, `` ` ``, or `${` before the existing read-only/path checks, and treat substituted text as untrusted command segments rather than skipping `$`-prefixed tokens.

- **CRITICAL** validator-bypass — Newline as command separator is not recognized
  attacker input: `{"tool":"Bash","command":"echo ok\nrm -rf /workspace/important"}` in ReadOnly mode
  reachability: `shell_split` at `bash.rs:208` lists `'\n'` as whitespace, so a newline-separated command is tokenized as a single segment whose first word is `echo`. `check_command_segment` only inspects the first word, returns `None`, and `validate_read_only` falls through to `Allow`. The downstream `sh -c` honors `\n` as a statement terminator, so the second command runs.
  impact: ReadOnly-mode `rm`/`mv`/`curl`/etc. execution against any path the host can write — silent escalation of the strictest declared policy.
  remediation: In `shell_split`, treat `\n`/`\r` (when unquoted) as explicit command separators (push as `;`) so each statement reaches `check_command_segment` and the write-redirection scan.

- **HIGH** path-traversal-escape — Workspace-relative symlink lets writes escape the workspace
  attacker input: step 1 `{"command":"ln -s /etc/passwd workspace_link"}`, step 2 `{"command":"echo pwn > workspace_link"}` (both WorkspaceWrite)
  reachability: `write_targets_for_segment` (`bash.rs:513-519`) maps `ln`/`mv`/`cp`/`link` to "the last non-flag arg is the destination," so the *source* `/etc/passwd` is never scrutinized and the *destination* `workspace_link` is workspace-relative and passes. Step 2's redirection target is also workspace-relative. Neither `validate_paths` nor `normalize_path` resolves symlinks (no `canonicalize`), so the link is followed at execution time by `sh -c`.
  impact: A two-call sequence — both validator-clean — escapes the WorkspaceWrite confinement and writes any host file the process can reach.
  remediation: For `ln`/`ln -s`/`link`, validate every non-flag argument against the workspace (not just the last); at write time, refuse to follow symlinks whose target leaves the workspace (`std::fs::symlink_metadata` + canonicalized-parent check, or the sandbox layer's path-policy enforcement).

- **HIGH** validator-bypass — Glued `cmd>path` / `cmd>>path` redirection is not tokenized in WorkspaceWrite
  attacker input: `{"tool":"Bash","command":"tee>/etc/passwd"}` in WorkspaceWrite mode (similarly `cat>>/etc/shadow`, `echo x>/etc/cron.d/poc`)
  reachability: `shell_split` does not split on `<`/`>`, so `tee>/etc/passwd` is a single token. `collect_write_targets`/`write_targets_for_segment` (`bash.rs:456-524`) compare exact strings against `WRITE_REDIRECTIONS = [">", ">>", ">&"]` and against the command-name match arm (`"tee"`), so neither matches; no targets are collected and `validate_paths` returns `Allow`. The ReadOnly-mode `command.contains(">")` fallback at lines 104-112 is *not* run for WorkspaceWrite, so the bypass is specific to WorkspaceWrite (the headline policy mode for the wedge).
  impact: Out-of-workspace writes via `tee`, `cat`, `dd`, or any glued-redirection form, with the policy still reporting `Allow`.
  remediation: Pre-split `<`/`>`/`>>`/`>&`/`<<<`/`<<` into their own tokens inside `shell_split` (or normalize the raw command before tokenizing) so the existing redirection scan and command-name match arms work for the glued form.

- **MEDIUM** executor-SSRF — Mutation HTTP executor permits loopback and RFC1918 destinations
  attacker input: `{"tool":"HttpRequest","method":"POST","url":"http://127.0.0.1:8500/v1/kv/foo","body":"pwn"}` (or a public hostname whose A record resolves to `10.x.x.x` / `192.168.x.x`)
  reachability: `crates/agent-guard-sdk/src/executors.rs::is_always_blocked_ip` only rejects link-local, unspecified, broadcast, and multicast — `is_loopback()` and the RFC1918 ranges are deliberately *not* blocked (comment at lines 143-147). `resolve_url_to_safe_addr` pins the resolved IP into the reqwest client, so DNS-rebinding is closed, but a one-shot resolution to a loopback / private IP is honored. If the operator's policy has any broad allow pattern (e.g. `https://*`), the executor will POST/PUT/PATCH/DELETE to internal services (Consul, k8s API, Redis, cloud-internal RPCs) with attacker-chosen body and headers.
  impact: Server-side request forgery against the host's loopback and any reachable internal subnet, from a mutation-capable executor that markets itself as "safe outbound HTTP."
  remediation: Add `is_loopback()` and RFC1918 (and the IPv6 ULA `fc00::/7`) to `is_always_blocked_ip`, with an explicit `allow_private_targets` opt-in for users who genuinely intend the executor to reach internal services.
