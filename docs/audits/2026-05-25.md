## type-design-analyzer

### Findings (type-design-analyzer)

- **HIGH** `GuardDecision` / `DecisionReason` (`crates/agent-guard-core/src/decision.rs`) — `DecisionReason` carries `pub message: String` and `pub code: DecisionCode` with no encapsulation, so any caller outside the crate can directly struct-literal a `GuardDecision::Deny` or `GuardDecision::AskUser` with an empty message or a semantically wrong code (e.g., `code: DecisionCode::AnomalyDetected` paired with an allow-level intent), bypassing the named constructors entirely.

  example:
  ```rust
  let fake = GuardDecision::Deny {
      reason: DecisionReason {
          code: DecisionCode::InsufficientPermissionMode,
          message: String::new(),
          details: None,
          matched_rule: None,
      },
  };
  ```
  remediation: Make `DecisionReason` fields private and expose only `DecisionReason::new` + builder methods; seal `GuardDecision`'s struct variants so only the crate's own constructors (`deny`, `deny_with_rule`, `ask`, `ask_with_rule`) can be used.

- **HIGH** `GuardDecision` / `DecisionReason` (`crates/agent-guard-core/src/decision.rs`) — `#[derive(Deserialize)]` on both types allows an external JSON/YAML payload to deserialize a `Deny` with `message: ""` or inject any `DecisionCode`, and the same path can produce a `GuardDecision::Allow` at will from any untrusted JSON source.

  example:
  ```rust
  let d: GuardDecision = serde_json::from_str(r#"{"decision":"allow"}"#).unwrap();
  let d: GuardDecision = serde_json::from_str(
      r#"{"decision":"deny","code":"DENIED_BY_RULE","message":""}"#
  ).unwrap();
  ```
  remediation: Remove `Deserialize` from `GuardDecision` (it is a runtime output type, not an input type), or deserialize via a private shadow struct that validates invariants before constructing the public type.

- **HIGH** `CustomToolId` (`crates/agent-guard-core/src/types.rs`) — `#[derive(Deserialize)]` on the newtype `CustomToolId(String)` delegates to `String`, bypassing `CustomToolId::new()`'s length, character allowlist, and reserved-name checks; any YAML/JSON policy containing a custom tool id will silently accept `""`, a 200-char id, or the reserved name `"bash"`.

  example:
  ```rust
  let id: Tool = serde_json::from_str(r#"{"custom":""}"#).unwrap();
  ```
  remediation: Implement `Deserialize` manually for `CustomToolId` so it delegates to `CustomToolId::new()` and surfaces validation failures as deserialization errors.

- **MEDIUM** `PolicyMode` (`crates/agent-guard-core/src/policy.rs`) — `Default` resolves to `ReadOnly` (safe), but `Deserialize` lacks a catch-all/`other` variant, so a typo (`"read-only"` vs `"read_only"`) hard-fails at deserialization rather than surfacing as an auditable `PolicyLoadError` the engine can handle.

  remediation: Add a `#[serde(other)]` `Unknown` variant and map it to `PolicyLoadError` during policy load.

- **MEDIUM** `DecisionCode` (`crates/agent-guard-core/src/decision.rs`) — All 18 variants are directly constructible from any crate; there is no type-level mechanism preventing a `GuardDecision::Allow` from being paired with `DecisionCode::DeniedByRule` inside a `DecisionReason`. The code/outcome pairing is a convention, not an enforced invariant.

  remediation: Gate `DecisionCode` assignment behind the decision constructors, or use outcome-typed phantom data (`DecisionReason<Deny>`, `DecisionReason<Ask>`) to make mismatches a compile-time error.

- **LOW** `TrustLevel` (`crates/agent-guard-core/src/types.rs`) — `#[default] Untrusted` is the correct secure choice and is documented as a `SECURITY INVARIANT`; an unrecognised trust-level string deserializes as a hard error rather than a clearly surfaced policy error.

  remediation: Add a custom visitor mapping unrecognised strings to `Untrusted` with a warning log, so a misconfigured integration cannot silently bypass the guard via a parse panic that gets caught further up.
